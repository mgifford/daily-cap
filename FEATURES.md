# Daily CAP: A Comprehensive Technical Feature Catalog

Daily CAP is a Canadian federal digital service quality benchmarking system that automates accessibility, performance, and usability scanning across high-traffic Government of Canada entry points. The project includes a complete pipeline for data ingestion, parallel scanning, impact aggregation, and accessible HTML report generation.

## Core Architecture

The system follows a linear workflow: traffic data → parallel scanning → metric aggregation → static report publishing. It processes approximately 130 top-traffic URLs daily through Lighthouse and axe-core scanners, extracting structured findings mapped to WCAG 2.2 requirements and Canadian bilingual obligations. English and French URLs are treated as separate scan targets throughout the pipeline.

## Key Components

**Configuration Management**: Central YAML configuration (`src/config/config.yaml`) controls scan parameters, URL limits, concurrency, multi-context scan targets, FPS prevalence rates, and severity weights. Validated at load time by `src/config/schema.js`. CLI overrides are applied via `src/config/loader.js`. `src/config/defaults.js` provides a lazy-loaded API for the rest of the pipeline.

**Traffic Data Ingestion**: Flexible adapter pattern accepts data from Canada.ca recent activity pages, Top Tasks lists, or static seed files. Multiple ingest sources are orchestrated through `src/ingest/orchestrate.js`. Falls back to seed data when live ingestion fails.

**Bilingual Inventory Building**: `src/inventory/build-inventory.js` normalises and deduplicates sources into a ranked inventory with `url_en` and `url_fr` fields where available. `src/inventory/ranking-engine.js` applies deterministic, tier-based ranking logic. Bilingual pairs are tracked through the full pipeline.

**Parallel Scanning**: `src/scanners/execution-manager.js` orchestrates concurrent Lighthouse and axe-core scans with configurable concurrency (default: 2), timeouts, and retry logic. Scans run across four contexts: desktop/mobile × light/dark.

**Impact Metrics**: `src/aggregation/impact-model.js` computes traffic-weighted quality impact using Statistics Canada Canadian Survey on Disability (CSD 2022) prevalence data. axe-core findings are mapped to CAN/ASC EN 301 549:2024 Functional Performance Statement (FPS) codes via `src/data/axe-to-fps.js`. All estimates are explicitly labelled as directional.

**HTML Report Generation**: All pages use semantic HTML5 landmarks, accessible tables with scoped headers, ARIA labels, skip-to-content links, and dark/light mode toggle (SVG moon/sun icons) with `localStorage` persistence. Generated by `src/publish/render-pages.js`.

**GitHub Actions Step Summary**: `src/cli/generate-summary.js` writes a Markdown step summary to `GITHUB_STEP_SUMMARY` (or stdout) after each scan. Includes aggregate scores, directional impact, FPS impact table, institution bilingual gap leaderboard, accessibility statement coverage, and top priority issues.

## Accessibility and Standards

Generated reports target WCAG 2.2 Level AA. The system produces bilingual parity analysis comparing English and French service quality. axe-core findings are mapped to WCAG success criteria and to Canadian FPS codes (CAN/ASC EN 301 549:2024 clauses 4.2.1–4.2.11). Each report includes policy context explaining why barriers matter to Canadians accessing federal services.

## Bilingual Analysis

Bilingual support is a first-class pipeline feature, not a post-processing step.

- Inventory records carry `url_en` and `url_fr` fields.
- English and French URLs are scanned as separate targets.
- `src/aggregation/bilingual-parity.js` computes EN/FR score gaps, flags missing counterparts, and produces a per-institution gap leaderboard sorted by mean absolute EN/FR accessibility score gap.
- Parity gaps are reported explicitly rather than silently averaged.
- Missing language counterparts are surfaced as a distinct finding type.
- Institution-level gap trend direction (improving / worsening / stable) is tracked across historical runs in `src/aggregation/institution-trends.js`.
- `details/bilingual-gap-leaderboard.json` is published with every report for downstream diffing and trend tracking.

## Canadian FPS Mapping

axe-core findings are mapped to CAN/ASC EN 301 549:2024 Functional Performance Statements (FPS) — the Canadian adaptation of European EN 301 549. The mapping covers ~80 axe rules across 11 FPS categories (4.2.1–4.2.11).

**Key differences from U.S. FPC categories:**
- FPS 4.2.9 (PST — Photosensitive seizure triggers) is Canadian-specific; maps `blink` and `marquee` rules
- FPS 4.2.11 (P — Privacy) has no disability population to model; excluded from prevalence estimates
- WVC (4.2.6) covers vocal capability more broadly than the U.S. equivalent
- LMS (4.2.7) combines manipulation and strength into one category
- LC (4.2.10) covers cognition, language, and learning

**Key files:**
- `src/data/axe-to-fps.js` — `AXE_TO_FPS` Map; `getFpsCodesForRule()`, `hasFpsMapping()` helpers; `Next review:` comment triggers bi-annual freshness check
- `src/data/fps-labels.js` — `FPS_CODES`, `FPS_LABELS`, `FPS_CLAUSES`, `FPS_DESCRIPTIONS`, `FPS_SVGS` (each SVG has `role="img"`, `<title>`, `<desc>`)
- `src/data/statistics-canada-disability-stats.js` — Canonical prevalence data (CSD 2022); per-FPS `rate`, `estimated_population`, `source_note`; `isStatsDataStale()`, `getFpsPrevalenceRates()` helpers

## Accessibility Statement Detection

`src/scanners/accessibility-statement-runner.js` and `src/aggregation/accessibility-statements.js` detect and assess accessibility statements on scanned pages.

Checks include:

- whether a statement exists
- whether it is linked clearly from the page
- whether contact information is present
- whether compliance status is mentioned
- whether alternatives or support paths are mentioned
- whether a date or freshness marker is present
- whether English and French variants both exist

Statement presence is reported as a governance and transparency signal. It does not indicate accessibility conformance.

## Platform and CMS Detection

`src/scanners/platform-fingerprint-runner.js` and `src/aggregation/platform-signals.js` detect CMS, design system, and hosting platform signals from scanned pages. Results feed into CMS and platform cohort breakdowns in reports.

## Institution Scorecards and Trends

`src/aggregation/institution-scorecards.js` groups scanned URLs by Government of Canada institution. Each institution receives an aggregated scorecard with missing FR count, missing statement count, high bilingual gap pair count, and priority issue scores.

`src/aggregation/institution-trends.js` maintains per-institution trend history across daily runs. Each snapshot now includes `mean_abs_accessibility_gap` and `mean_abs_performance_gap` (from bilingual pairs for that institution). A `parity_trend` field (`improving` / `worsening` / `stable` / `insufficient_data`) and `parity_gap_delta` compare earliest vs latest gap magnitude across all historical reports.

Individual institution pages are published at `docs/reports/daily/YYYY-MM-DD/details/institutions/`.

## Barrier History Tracking

`src/aggregation/barrier-history.js` maintains a rolling 30-day view of key barrier signals across daily runs:

- missing French counterparts
- missing or low-quality accessibility statements
- high EN/FR accessibility score gaps

Results are rendered as trend charts in each daily report.

## Trend Analysis and Regression Alerts

`src/aggregation/trend-analysis.js` compares the current run against the previous report. Day-over-day regressions on key metrics trigger alerts in the published report. Historical data spans up to 30 days.

## Priority Issue Tracking

`src/aggregation/priority-issues.js` identifies and ranks the highest-impact recurring accessibility barriers across scanned URLs. Issues are prioritised by estimated reach and severity.

## Cohort Quality Analysis

`src/aggregation/cohort-quality.js` computes quality metrics by service-pattern cohort (sign-in, forms, dashboards, etc.) and by CMS or platform type.

## Operational Automation

GitHub Actions workflows handle daily scans with automatic report publishing to GitHub Pages. Reports older than 14 days are archived by `src/publish/archive-reports.js` while retaining core JSON data.

**Workflows:**

| Workflow | Trigger | Purpose |
| :--- | :--- | :--- |
| `daily-scan.yml` | Daily schedule + `workflow_dispatch` | Run full pipeline; publish report to GitHub Pages |
| `scan-self-accessibility.yml` | Push to main (reports changed) + monthly | axe-core scan of published GitHub Pages with `self-accessibility` label; skips if open issues exist |
| `check-data-freshness.yml` | March 20 + September 20 + `workflow_dispatch` | Check Statistics Canada data and axe-to-FPS mapping review dates; opens `data-review` issue if stale |

## Command-Line Interface

Main pipeline (`run.js`):

```bash
node run.js --date YYYY-MM-DD --limit N --mode live|mock --output-root PATH --source-mode live|seed
```

| Flag | Default | Description |
| :--- | :--- | :--- |
| `--date` | Today | Run date in YYYY-MM-DD format |
| `--limit` | 130 | Maximum URLs to scan |
| `--mode` | mock | `live` runs real scans; `mock` uses fixture data |
| `--output-root` | `./` | Root directory for output |
| `--source-mode` | live | `live` ingests from Canada.ca; `seed` uses static seed |

Summary generator (`src/cli/generate-summary.js`):

```bash
node src/cli/generate-summary.js [--output-root PATH] [--date YYYY-MM-DD]
# or: npm run summary
```

## HTML Report Features

### Daily report page sections

- **Header**: Run date, scan summary, bilingual coverage count
- **Bilingual parity**: EN/FR pair coverage, parity gap distribution, institution gap leaderboard with trend direction, missing counterparts
- **Accessibility impact**: Traffic-weighted barrier estimates by FPS disability category (CAN/ASC EN 301 549:2024) using Statistics Canada CSD 2022 prevalence rates
- **Accessibility statements**: Coverage rate, quality signal breakdown, EN/FR statement parity
- **Performance summary**: Core Web Vitals signals, mobile vs. desktop comparison
- **Top URLs**: Ranked table with per-URL scores, language, institution, trend arrows
- **Detailed findings**: Expandable axe-core findings with WCAG mapping
- **Platform and CMS breakdown**: Distribution by detected CMS and design system
- **Institution scorecards**: Per-institution aggregated scores with parity trend indicators
- **Barrier history chart**: 30-day rolling view of key barrier signals
- **Priority issues**: Highest-reach recurring barriers
- **Trend charts**: Historical score trends across the 30-day window
- **Methodology notes**: Explicit caveats on automated signal limitations

### Institution trend pages

Each institution has a dedicated trend page at `details/institutions/{slug}.html` showing:

- Latest mean accessibility score and EN/FR accessibility gap
- Parity trend direction (improving / worsening / stable / insufficient data)
- Trend charts for accessibility score, EN/FR gap, missing FR counterparts, and missing statements
- Full historical data table with gap columns

### Dashboard page

- Report history index with date links
- Download links for archived reports (zip)
- Latest aggregated scores

### Self-compliance

Daily CAP reports are scanned for WCAG 2.2 AA conformance monthly via the `scan-self-accessibility.yml` workflow using `github/accessibility-scanner@v3`. Open issues must be resolved before new scans are triggered. The project benchmarks accessibility; its own outputs must meet or exceed the same standards it measures.

## Methodology Guardrails

Daily CAP reports use precise language to avoid overclaiming:

| Avoid | Use instead |
| :--- | :--- |
| "certified accessible" | "automated signal" |
| "measured affected users" | "directional estimate" |
| "all Government of Canada services" | "benchmark coverage" |
| "compliant statement" | "detected statement" |
| "proven barrier-free" | "no automated barriers detected" |
| "complies with EN 301 549" | "mapped to FPS category" |

## Adapting for Other Jurisdictions

Daily CAP is designed for the Canadian federal context. Key adaptation points for other jurisdictions:

- **Traffic data source**: Replace Canada.ca recent activity with the local analytics source
- **Disability prevalence data**: Replace `src/data/statistics-canada-disability-stats.js` with jurisdiction-specific data; update `src/config/config.yaml` `prevalence_rates` to match
- **Accessibility standards mapping**: Update `src/data/axe-to-fps.js` for the local standard; update FPS labels in `src/data/fps-labels.js`
- **Domain registry**: Update institution detection logic in `src/aggregation/institution-scorecards.js`
- **Bilingual requirements**: Adjust or disable if the jurisdiction is not officially bilingual
- **Seed inventory**: Replace `src/ingest/canada-seed.js` with a jurisdiction-specific seed list
